‚óŹ Looking at the workflow, I need to think about the architecture. Cloud
  Workflows YAML can't execute Python, so the manifest has to be computed by a
  Cloud Run Job and written somewhere the Workflow can read it. Let me design
  this:

  1. New JOB_TYPE plan_grid_point_backfill: reads cursor, computes manifest,
  writes it to GCS (does NOT persist cursor yet).
  2. Workflow reads the manifest from GCS, iterates batches with sleep gaps
  between Cloud Run Job invocations.
  3. Final JOB_TYPE commit_grid_point_backfill: promotes the saved manifest's
  next_cursor to the current cursor.

  This way, if the workflow fails mid-run, the cursor stays put and tomorrow
  retries the same range.
