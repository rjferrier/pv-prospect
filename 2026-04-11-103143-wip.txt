‚óŹ Good exploration. Here's my assessment of your proposal:

  The individual-files-then-consolidate approach is the right call. GCS doesn't
  support append, so a shared log file would require read-modify-write which is
  inherently racy. Individual files with atomic creates is the correct pattern.

  Refinements I'd suggest:

  1. Filename collisions ‚ÄĒ <HHMMSS> is too coarse. Two workers can easily write
  in the same second. I'd use <HHMMSS.ffffff> (microsecond timestamp) or
  <HHMMSS>-<short-uuid>. The former keeps files naturally sorted by time.
  2. Where to hook it ‚ÄĒ All writes flow through FileSystem.write_text() /
  write_bytes() (in pv-prospect-etl). A decorator wrapping FileSystem is the
  cleanest approach: every write through the decorated filesystem also emits a
  log entry. This captures both Loader-mediated writes and direct
  fs.write_text() calls (like the transformation module's write_csv helper).
  You'd wire it up at the factory level.
  3. What the decorator should NOT log ‚ÄĒ The log entries themselves. The
  decorator needs to skip writes under the logs/ prefix to avoid infinite
  recursion.
  4. Consolidation ‚ÄĒ A simple step at the end: list all files under
  logs/<YYYY-MM-DD>/, read each one, concatenate, write to
  logs/<YYYY-MM-DD>.txt, delete the individual files. This could be a small
  Cloud Run job or an extra step in the workflow.
  5. Scope ‚ÄĒ Should manifest/cursor writes (backfill state) also be logged, or
  only data files? Those feel like internal bookkeeping rather than pipeline
  outputs.

  Want me to implement this? The plan would be:
  - Add a LoggingFileSystem decorator in pv-prospect-etl that wraps any
  FileSystem
  - Wire it into the factory
  - Add a consolidation function
  - Add the consolidation as a workflow step in Terraform

     Plan: Write-logging for extraction and transformation workflows

     Context

     We want to track every data file written to the staging bucket during
     workflow runs. Each concurrent worker writes an individual log entry file
     to avoid race conditions (GCS doesn't support append). A consolidation step
      at the end of each workflow merges entries into a single log file per
     workflow run.

     Log entry path (intermediate):
     logs/<YYYY-MM-DD>/<workflow-name>/<HHMMSS.ffffff>.txt
     Consolidated log path:
     logs/<YYYY-MM-DD>/<YYYY-MM-DD-HHMM>-<workflow-name>.txt

     Only data file writes are logged (not manifest/cursor writes to
     resources_fs).

     Approach

     1. New LoggingFileSystem decorator (pv-prospect-etl)

     New file: pv_prospect/etl/storage/logging_fs.py

     LoggingFileSystem wraps a FileSystem and holds a reference to a separate
     log FileSystem. On write_text/write_bytes, it delegates to the inner FS
     then writes a single-line log entry to the log FS.

     Constructor: __init__(self, inner: FileSystem, log_fs: FileSystem,
     workflow_name: str, label: str)

     - label is the storage prefix (e.g. 'raw', 'cleaned') so the logged path is
      fully qualified within the bucket.
     - All other FileSystem methods delegate to inner without logging.
     - Log entry content: <ISO-timestamp> <label>/<relative-path>

     Same file also contains consolidate_logs(log_fs, workflow_name, date)
     which:
     1. Lists files under <YYYY-MM-DD>/<workflow-name>/
     2. Reads and sorts by timestamp
     3. Writes <YYYY-MM-DD>/<YYYY-MM-DD-HHMM>-<workflow-name>.txt
     4. Deletes the individual entry files

     2. Config changes

     pv-prospect-etl/pv_prospect/etl/resources/config-default.yaml ‚ÄĒ add:
     log_storage:
       backend: 'gcs'
       bucket_name: 'pv-prospect-staging'
       prefix: 'logs'

     DataExtractionConfig and DataTransformationConfig ‚ÄĒ add optional
     log_storage: AnyStorageConfig | None field. When absent/null, no
     write-logging occurs (local dev stays unaffected).

     3. Entrypoint changes

     Extraction entrypoint (pv-prospect-data-extraction/.../entrypoint.py):
     - Read WORKFLOW_NAME env var
     - If WORKFLOW_NAME is set and config.log_storage is not None, wrap
     staging_fs with LoggingFileSystem(..., label='raw')
     - Add consolidate_logs job type that calls the consolidation function

     Transformation entrypoint
     (pv-prospect-data-transformation/.../entrypoint.py):
     - Read WORKFLOW_NAME env var
     - If set and config.log_storage is not None, wrap cleaned_fs
     (label='cleaned'), batches_fs (label='prepared-batches'), and prepared_fs
     (label='prepared') with LoggingFileSystem
     - Add consolidate_logs transform step

     4. Terraform workflow changes

     All four workflows (extract, transform, pv-site-backfill,
     weather-grid-backfill):
     - Assign workflow_name in the init step
     - Add WORKFLOW_NAME env var to every containerOverrides block
     - Add a consolidate_logs step before done (calls the same Cloud Run job
     with JOB_TYPE=consolidate_logs or TRANSFORM_STEP=consolidate_logs)

     5. Exports

     Update pv_prospect/etl/storage/__init__.py to export LoggingFileSystem and
     consolidate_logs.

     6. Tests

     New test directory: tests/unit/storage/logging_fs/

      not, label is prepended
     - test_consolidate_logs.py ‚ÄĒ verify entries are merged, sorted, and
     individual files deleted
