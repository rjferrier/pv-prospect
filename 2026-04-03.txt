вқҜ The OpenMeteo weather API allows up to c. 10,000 calls per day, and each call
 may cover up to 14 days of weather history per location. With this daily call

2. Choose one of the `sample_*.csv` files in `uk-geo/point_samples/` and for

3. Use the remaining budget for querying weather data further back in time.
Eventually I would like to build up several years of historical data. We can do
 this by starting 14 days ago and working our way backwards, like so: (a)
choose the sample file *preceding* the one we chose in Step 2; (b) query the
last 14 days for each location in that file; (c) go back another 14 days; (d)
choose the sample file preceding the one we chose last time; repeat steps (b)
through (d). We should be able to do this eight times before hitting the budget
 (8 * 1,066 = 8,528), which means we'll amass 112 days of weather data (14 * 8)
 in a single day of API calls.

Does this make sense as a strategy and how would we encode and automate it?
Suggestions:
* For Step 2, we choose the sample file according to the current date. If D is
the date in terms of epoch days (days since 1970-01-01), and there are N sample
 files, we simply choose sample file # `D % N`.
* For Step 3, we create a manifest file that lists the sample files to be used
and which dates to be queried for each. Each row jumps backwards by 14 days,
and the sample file is chosen based on the coverage end date and the modulus
rule above. It means we won't end up with the spatial density of data going
forward (Step 2), but we will never have a gap of one or more days in the
weather history.

as one 'block'. We then go back another three years to complete another block.
At this point we might want to start increasing the density of the near-term
data, so we might start another block (Block 3) that covers approximately the
same time range as Block 1, but is staggered in time and uses a different
sample file, thus doubling the density of data. Then Block 4 might go back
another three years atop Block 2; Block 5 doubles the density of Block 3; etc.
(building a triangle of blocks). However, this is a longer-term idea. The point
 is that the modulus rule, and the scattering pattern of points across sample
files, keeps things flexible and ensures a nice spread of data regardless of
spatial density.
* Perhaps the service triggered by the scheduler could also check whether the
last batch of weather data files have been written to the bucket as expected.
If they have not, it would try and process the failed batch first.
